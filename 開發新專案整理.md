# 開發新專案整理

這份清單用來協助 .NET/C# 新專案完成基礎初始化、開發規範設定、AI 工具輔助與專案收尾文件整理。建議依序執行，並在每個階段確認產出的檔案與目前專案需求一致。

## 一、專案初始化步驟

### 1. 建立 `.gitignore`

在專案根目錄執行：

```bash
dotnet new gitignore
```

這會產生適用於 .NET 專案的 `.gitignore`，避免將 `bin/`、`obj/`、本機設定與暫存檔提交到版本控制。

### 2. 建立 `.editorconfig`

在專案根目錄執行：

```bash
dotnet new editorconfig
```

`.editorconfig` 用來統一縮排、換行、命名規則與 C# 程式碼風格。產生後請先檢查內容，再依專案需求補上共用規則。

完成後建議執行一次格式化或重新儲存主要檔案，確認編輯器已套用新的設定。

## 二、開發新專案基本準則

### Model / DTO 驗證規則

- 對外接收資料的 Model、Request DTO、ViewModel，應明確加上 `DataAnnotations` 驗證屬性。
- 常見屬性包含 `Required`、`StringLength`、`Range`、`RegularExpression`、`EmailAddress`。
- 驗證規則應反映實際業務限制，不要只為了通過檢查而加入空泛標記。
- 若是 EF Core Entity，資料庫欄位限制與關聯設定可視情況交由 Fluent API 補強，避免把資料庫映射細節全部塞進資料模型。

### 註解與可讀性

- Public API、主要解題函式與容易被重複使用的方法，應加上 XML documentation comments。
- 註解重點放在「為什麼這樣做」與「演算法或流程的關鍵判斷」，避免逐行描述程式碼已經表達清楚的內容。
- 命名優先於註解。若必須靠大量註解才能理解，應先檢查方法切分與命名是否足夠清楚。

## 三、VS Code 擴充套件與工具

### Commander 擴充套件

- **用途**：在 Copilot Chat 或聊天室視窗中呼叫 `@commander`，用自然語言描述要執行的指令。
- **使用方式**：描述需求後，讓工具協助組合或執行對應命令。
- **適合場景**：查詢可用命令、整理重複性操作、快速觸發專案常用工作。
- **快捷鍵**：按下 `F1` 可開啟命令面板並列出可用指令。

### Mermaid 圖表工具

- **用途**：使用 `@mermaid` 產生流程圖、時序圖、類別圖、架構圖等 diagram。
- **使用方式**：先用文字描述流程或關係，再請 AI 轉成 Mermaid 語法。
- **適合場景**：整理需求流程、說明系統架構、補充 README 或技術文件中的視覺化說明。

## 四、專案收尾與後續步驟

在準備提交或交付前，依序完成以下檢查：

1. 在 `Main` function 或主要進入點加入可執行的測試資料。若 XML 註解是題目描述或原始需求，請不要異動。
2. 為每個主要 function 補上 XML `summary`，說明用途、解題概念、輸入條件與輸出結果。
3. 在關鍵演算法或判斷流程旁加入必要註解，說明解題思路，不要加入無意義的逐行註解。
4. 參考 `docs/readme-template.md` 建立或整理 `README.md`。
5. `README.md` 應包含題目說明、限制條件、解題概念與出發點、每一種解法的設計說明，以及每一種解法的範例演示流程。
6. 完成後執行專案可用的建置、測試或範例執行命令，確認 README 中記錄的指令與實際結果一致。
7. 最後執行 `git diff --check`，確認沒有多餘空白或換行問題。
8. 最終確認都沒問題就直接 auto commit git push, commit message 採用 Conventional Commits 格式
